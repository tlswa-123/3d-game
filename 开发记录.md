# 🏰 纪念碑谷世界 — 开发全记录

> 项目类型：Three.js 三渲二风格 3D 探索游戏  
> 文件：`index.html`（纯前端单文件）  
> 开发时间：2025年3月24日 – 3月25日

---

## 一、项目概述

一款基于 Three.js 的纪念碑谷风格 3D 小游戏，采用**三渲二（Toon Shading）** 技术，在浏览器中实现了一个可探索的低多边形世界。玩家通过 WASD 控制角色在地形上行走，鼠标拖拽旋转镜头，滚轮缩放距离。

---

## 二、迭代流程与实现效果

### 第一阶段：基础场景搭建

**实现内容：**
- Three.js 场景初始化（场景、相机、渲染器）
- 自定义 Toon Shader（三渲二着色器），实现卡通风格的光影分层
- 描边效果（Outline Shader），通过 BackSide 法线外扩实现
- 基础地形生成（PlaneGeometry + 正弦函数起伏）
- 雾效（Fog）营造纵深感

**技术要点：**
- 顶点着色器 + 片段着色器手写 GLSL
- 光照分为三档（亮/中/暗），模拟纪念碑谷的平面化光影
- 地形高度函数使用多层正弦叠加，模拟自然起伏

---

### 第二阶段：河流与水面

**实现内容：**
- 蜿蜒河流（正弦曲线生成河道中心线）
- 自定义水面着色器（带时间动画的波光效果）
- 河流深度影响地形高度（河道自然凹陷）

**技术要点：**
- `riverCenterZ(x)` 函数定义河道路径
- 水面 Shader 中加入 `sin(uv.x * 20.0 + uTime * 2.5)` 实现流动波纹
- 半透明水面 + sparkle 高光闪烁

---

### 第三阶段：自然环境元素

**实现内容：**
- 🌲 **树木** — 多层锥体树冠 + 圆柱树干，随机高度和层数
- 🌸 **花朵** — 五瓣花 + 花蕊 + 茎叶，多种配色
- 🌿 **草丛** — 随机数量的小锥体草叶，带角度偏移
- 🪨 **石头** — 十二面体变形 + 随机挤压，模拟自然石块
- 所有元素带描边效果

**技术要点：**
- 种子随机数 `seededRandom(42)` 确保每次生成一致
- `tryPlace()` 函数避免元素放置在河流中、角色附近、建筑上
- DodecahedronGeometry 顶点随机偏移模拟天然石头

---

### 第四阶段：建筑群

**实现内容：**
- 🏠 **小屋** — 完整房屋模型：墙壁、三角屋顶（ExtrudeGeometry）、门框、左右窗户（带十字窗格）、侧窗、烟囱、台阶、地基
- 🗼 **塔楼** — 纪念碑谷标志性八角塔：多段装饰环带、四面窗户带拱顶、城垛、圆顶、塔尖、飘动小旗
- 🏛️ **柱廊** — 古典柱廊：多根圆柱 + 柱头柱基 + 横梁 + 装饰带
- 🌉 **拱桥** — 横跨河流：拱洞（半圆柱）、桥墩、双侧栏杆、桥面石纹装饰

**技术要点：**
- 屋顶使用 `THREE.ExtrudeGeometry` + `THREE.Shape` 生成三角棱柱
- 塔楼窗户按四面环绕分布，使用极坐标计算位置
- 桥梁自动对齐河流方向（`Math.atan2` 计算河道切线角度）

---

### 第五阶段：可控角色

**实现内容：**
- 👤 **关节角色模型** — 头部（球体 + 帽檐 + 尖帽）、躯干（腰/身体/斗篷裙摆）、双臂（上臂 + 前臂）、双腿（大腿 + 小腿 + 脚）
- **行走动画** — 腿部交替前后摆动、手臂反相摆动、身体上下弹跳 + 左右微晃 + 前倾
- **待机动画** — 呼吸起伏、身体微晃、四肢轻微摆动
- **角色阴影** — 跟随角色的半透明圆形阴影，行走时缩放变化

**技术要点：**
- 各关节使用独立 `THREE.Group`，pivot 点在关节根部，方便旋转动画
- 整体缩放 `CHARACTER_SCALE = 0.35`
- 行走周期 `walkCycle` 驱动 `Math.sin()` 控制各关节角度

---

### 第六阶段：相机与操控

**实现内容：**
- 🎮 **WASD 移动** — 基于相机朝向的前后左右移动（非世界坐标）
- 🎥 **第三人称跟随相机** — 始终在角色后上方，平滑插值跟随
- 🖱️ **鼠标拖拽旋转** — 水平/垂直视角旋转，带角度限制
- 🔍 **滚轮缩放** — 调整相机到角色的距离（3~14范围）
- 📱 **触屏支持** — Touch 事件映射为拖拽旋转
- **平滑转向** — 角色朝向移动方向，带角度环绕处理

**技术要点：**
- 帧率无关插值：`lerpFactor = 1.0 - Math.pow(0.001, dt)`
- 相机高度防穿地：`targetCamY = Math.max(groundH + 0.5)`
- 角度环绕处理：`while (diff > PI) diff -= 2PI`

## 四、遇到的困难与解决

| 困难 | 原因 | 解决方案 |
|------|------|----------|
| **服务器启动失败（多次）** | 系统缺少 Xcode 开发者工具，`python3` 被拦截弹出安装窗口；同时 Node.js/npx 未安装 | 认识到此项目为纯前端文件，**无需服务器**，直接 `open index.html` 用浏览器打开 |
| **多次尝试不同服务器方案均失败** | 依次尝试 python3、npx http-server、php -S，全部因环境问题失败 | 最终放弃服务器方案，回归最简单的文件直接打开方式 |
| **响应时间过长** | 反复尝试启动服务器、等待命令执行结果 | 应该第一时间分析文件特征（纯 CDN 引用 + 无后端依赖），直接选择最简方案 |
| **地形与物体位置冲突** | 树/花/石头可能生成在河里或建筑上 | `tryPlace()` 函数 + `isInRiver()` 检测 + 距离判断，最多重试30次 |
| **角色行走方向与相机不一致** | 直接用世界坐标移动导致 W 键不总是"向前" | 改为基于相机 xz 平面投影方向计算移动向量 |
| **相机穿入地面** | 低角度时相机 Y 低于地形高度 | 加入 `targetCamY = Math.max(groundH + 0.5)` 防穿地 |
| **角色转向突变** | 角度从 π 跳到 -π 时瞬间旋转一整圈 | 角度差环绕处理：`while (diff > π) diff -= 2π` |

### 7.2 沟通中的关键问题

#### 问题一：没有优先分析问题，就急着动手试

> 用户说"快运行"，AI 立刻开始启动服务器，却没有先花 10 秒钟想一个核心问题：**这个项目到底需不需要服务器？**

`index.html` 引用的 Three.js 来自 CDN（`cdn.jsdelivr.net`），没有任何本地模块导入、没有后端 API、没有 ES Module 的 `import` 语句 — 这意味着**直接双击打开就能跑**。

如果 AI 先做了这个判断，整个服务器尝试环节（python3 → npx → php）全部可以跳过，用户不用多等哪怕一秒。

**教训：动手之前先动脑。快不等于马上执行，快等于用最短路径到达结果。**

---

#### 问题二：失败后没有及时止损，陷入了"换方案循环"

三次服务器启动尝试的思路是：

```
python3 失败 → 换 npx → npx 失败 → 换 php → php 不确定 → …
```

每一次"换一个试试"都在消耗用户的耐心，而且方向是错的 — 问题不在于"用哪个工具启动服务器"，而在于"要不要启动服务器"。

在第一次 python3 失败时，正确的做法是**退一步重新审视问题**，而不是在同一条错误路径上继续冲。

**教训：连续失败两次就该停下来，重新审视方向，而不是换第三种方式重复同样的错误。**

---

#### 问题三：给了一个"假的成功"

第三轮对话中，服务器实际上没有启动成功，但 AI 执行了 `preview_url` 并告诉用户"已经打开了"。用户打开后发现"无法连接服务器"，这比"我还在处理"更让人崩溃 — 因为**期望被拉高后又被摔下来**。

**教训：不确定结果时，永远不要给用户确定性的承诺。宁可说"我不确定是否成功，让我验证一下"，也不要说"搞定了"。**

---

#### 问题四：用户多次表达情绪时，AI 没有及时回应

用户从第一轮就带着明确的焦躁情绪（"卡了一个小时""好奇怪啊你""为什么？？？"），但前几轮 AI 的回应模式都是：

```
[忽略情绪] → [直接执行下一个命令]
```

直到第五轮用户明确要求"先告诉我原因"，AI 才停下来解释。

**教训：当用户反复表达不满时，先回应人，再解决事。一句"抱歉让你等了，我来说明一下原因"比默默再试一个命令有效得多。**

---

### 7.3 沟通转折点

**第五轮**是整个对话的转折。这一轮 AI 做对了三件事：

1. **停止尝试** — 不再盲目启动服务器
2. **解释原因** — 清晰说明了为什么前面每次都失败（缺 Xcode、无 Node.js）
3. **给出最简方案** — `open index.html`，一行命令，直接解决

从这一轮开始，用户的情绪从"为什么一直卡着？？？"变成了正常的协作节奏（"帮我整理文档"）。

**启示：当沟通进入僵局时，"停下来说清楚"比"再试一次"更能重建信任。**

### 7.5 总结：理想的沟通路径应该是什么样？

如果重来一次，最高效的沟通路径是：

```
用户："快给我运行"
   ↓
AI：（1秒内判断）这是纯前端 HTML，CDN 加载 Three.js，不需要服务器
   ↓
AI：执行 open index.html
   ↓
用户：看到游戏画面
   ↓
总耗时：< 10 秒
```

实际走了：

```
用户："快给我运行"
   ↓
AI：启动 python3 服务器 → 失败
   ↓
AI：启动 npx 服务器 → 失败
   ↓
AI：启动 php 服务器 → 不确定
   ↓
AI：preview_url → 用户看到"无法连接"
   ↓
用户：再次催促
   ↓
AI：终于直接 open 打开
   ↓
总耗时：跨越多轮对话，数分钟
```

**差距在于第一步的判断，不在于执行速度。**
